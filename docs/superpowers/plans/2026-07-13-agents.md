# redbar Agents — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Specs:** `docs/superpowers/specs/2026-07-12-design.md` + `docs/superpowers/specs/2026-07-13-multi-language-and-agents.md`

**Depends on:** `2026-07-13-engine.md` — specifically `inspect(root)` returning ranked `Gap[]` with a `kind` (unit/integration/e2e) on each. **Do not start this plan before the engine's day-4 checkpoint passes on a real repo.** If the engine's gaps are wrong, everything here writes tests for the wrong holes.

**Goal:** Turn a ranked gap into a passing test file, in any language, through any agent CLI the developer already has installed. Plus `redbar init`, which proposes the test libraries for the detected language and lets the human approve them.

**Architecture:** The specialist test agents are **not code** — they are `conventions/<language>/<kind>.md`, the house testing standard. A worker composes `convention + gap + source` into a prompt and `spawn()`s the agent CLI the developer already has (Claude Code, Codex, Copilot, or anything else via config). No SDK, no API key, no new inference cost. A worker pool runs N gaps at once, partitioned by target file so two workers can never touch the same file. Every worker runs the test it just wrote: green stays, red retries once, red again is marked `needs-human` and never committed.

**Tech Stack:** Node 24, TypeScript 7.0.2, Vitest 4. Still zero runtime dependencies — `node:child_process` is the whole LLM integration.

## Global Constraints

- **English only** — code, identifiers, comments, tests, and the conventions documents.
- **Zero runtime dependencies.** `dependencies` stays empty. The agent adapter is `spawn()`, not an SDK.
- **Zero new inference cost.** redbar never holds an API key. It drives the CLI the developer is already licensed for.
- **The tool never installs a dependency on its own.** `init` prints the diff and the exact command; the human presses enter. Touching someone's `package.json`/`pom.xml`/`Cargo.toml` unasked is a rejected PR at best and a supply-chain incident at worst.
- **Never leave a red test in the repository.** A failing test that leaks into the demo erases everything else.
- **Adding a language is still one line** in `languages.ts` plus a conventions markdown. No `switch (lang)` anywhere.
- **Commits:** conventional, lowercase, no trailing period. No `Co-Authored-By`, no emoji, no mention of Claude/AI/Anthropic.

## File Structure

| File | Responsibility |
|---|---|
| `conventions/<lang>/{unit,integration,e2e}.md` | **the house standard.** The real deliverable — the specialist agents ARE these files |
| `src/conventions.ts` | load the convention for (language, kind); fail loudly when it does not exist |
| `src/agents.ts` | the multi-LLM adapter — `spawn()` a CLI, auto-detect from `PATH`, config escape hatch |
| `src/prompt.ts` | `convention + gap + source` → the prompt string. Pure |
| `src/testpath.ts` | source file + language + kind → where its test file belongs |
| `src/fix.ts` | worker pool, file partitioning, the quality gate |
| `src/init.ts` | detect the language, propose its test libs, print the command. Never installs |
| `src/report.ts` | `.redbar/gaps.json` (for the agent) + `REDBAR.md` (for the human and the PR) |

### Task 1: `conventions/` — the library's canonical standard

> **Reversal — read `docs/superpowers/specs/2026-07-13-conventions-are-library-standards.md` before starting.** The original design called this "*this company's* testing standard", to be authored by the team. That is wrong and this plan no longer does it. The complaint is "everyone writes tests their own way"; writing a house standard does not fix that, **it adds a seventh way**. The convention for a layer is **the canonical standard of the library that runs it** — Playwright's own best-practices page, the Vitest docs, the pytest idiom, the JUnit 5 user guide.
>
> **Nobody argues with the Playwright docs in a code review. Everybody argues with the standard a colleague invented last week.**
>
> Two practical consequences for you, the implementer:
> - **Do not invent guidance.** Every rule in these files must be traceable to the library's official documentation. Put the source URL at the top of each file. If you cannot cite it, do not write it.
> - The project can override with `.redbar/conventions/<lang>/<layer>.md`, which is **appended** to the shipped standard. That is for genuinely local choices (MSW vs nock, which fixture factory) — not for restyling the standard.

**Files:**
- Create: `conventions/ts/{unit,integration,e2e}.md`
- Create: `conventions/java/{unit,integration,e2e}.md`
- Create: `conventions/python/{unit,integration,e2e}.md`
- Create: `conventions/rust/{unit,integration,e2e}.md`
- Create: `conventions/php/{unit,integration,e2e}.md`
- Create: `src/conventions.ts`
- Test: `test/conventions.test.ts`

**Interfaces:**
- Produces:
```ts
export function loadConvention(lang: Language, kind: TestKind, root?: string): string
// throws: `redbar: no convention for <lang>/<kind>. Write conventions/<lang>/<kind>.md — the
//          standard is the deliverable, not the tool.`
```

**Every convention document answers exactly these five questions, in this order.** The structure is fixed so the prompt builder can rely on it, and so a human can diff two languages' standards side by side:

1. **Where the test file lives** (exact path rule, with an example).
2. **What one test looks like** (a complete, copy-pasteable example — the *shape*, not a lecture).
3. **What to assert** (behavior, boundaries, the error path — and what NOT to assert).
4. **What to mock, and what never to mock** (the line between unit and integration lives here).
5. **Naming** (test name, file name, describe/class name).

- [ ] **Step 1: Write the TypeScript unit convention**

`conventions/ts/unit.md`:
```markdown
# TypeScript — unit tests

## Where it lives
Next to the source file, `.test.ts` suffix: `src/order/total.ts` → `src/order/total.test.ts`.

## What one test looks like
    import { describe, expect, it } from 'vitest'
    import { total } from './total.js'

    describe('total', () => {
      it('sums the line items', () => {
        expect(total([{ price: 10, qty: 2 }])).toBe(20)
      })

      it('throws on a negative quantity', () => {
        expect(() => total([{ price: 10, qty: -1 }])).toThrow(/negative/)
      })
    })

## What to assert
The behavior the function promises, at its boundaries: the zero case, the empty case,
the error path. One behavior per `it`. Assert the value, not the number of calls.

Do NOT assert on internals — a test that breaks when you rename a private variable is a
test that will be deleted within a month.

## What to mock
Nothing. A unit test that needs a mock is telling you the function has an I/O dependency
it should not have — either the function is wrong, or this is an integration test.
The only exception is time and randomness: `vi.setSystemTime`, and inject the seed.

## Naming
`describe` = the symbol under test, exactly as exported. `it` = the behavior, in the
present tense, no "should": `it('sums the line items')`, not `it('should sum items')`.
```

- [ ] **Step 2: Write the remaining TypeScript conventions**

`conventions/ts/integration.md` — same five headings. Key content: the test lives in `test/integration/`, it talks to a **real** dependency (a container, a temp DB, a local HTTP server), it mocks **nothing** below the boundary it is testing, and it cleans up after itself. Assert the round trip: write, read it back, and prove the shape survived.

`conventions/ts/e2e.md` — same five headings. The test lives in `e2e/`, uses `@playwright/test`, drives the app through the UI or the HTTP surface a real client uses, and asserts **what the user sees**, not what the server logged. Select by role and accessible name (`getByRole('button', { name: 'Checkout' })`), never by CSS class — a class rename is not a regression, and a test that says it is will get muted.

- [ ] **Step 3: Write the java, python, rust and php conventions**

Same five headings, same shape, translated to each ecosystem's idiom:

| Language | unit | integration | e2e |
|---|---|---|---|
| java | JUnit 5 + Mockito, `src/test/java/**/<Class>Test.java` | `@SpringBootTest` + Testcontainers | Playwright for Java |
| python | pytest, `tests/test_<module>.py`, plain `assert` | pytest + httpx / testcontainers | pytest-playwright |
| rust | `#[cfg(test)] mod tests` in the same file | `tests/` directory (integration by Cargo's own definition) | Playwright driving the built binary |
| php | PHPUnit, `tests/Unit/<Class>Test.php` | PHPUnit + a real DB, `tests/Integration/` | Playwright |

**Do not invent a house style per language.** Each document says the same thing the TS one says — assert behavior, mock nothing in unit, name for the reader — expressed in that language's idiom. The consistency IS the standard.

- [ ] **Step 4: Write the failing test for the loader**

`test/conventions.test.ts`:
```ts
import { describe, expect, it } from 'vitest'
import { loadConvention } from '../src/conventions.js'
import { LANGUAGES, byId } from '../src/languages.js'
import type { TestKind } from '../src/types.js'

const KINDS: TestKind[] = ['unit', 'integration', 'e2e']

describe('loadConvention', () => {
  it('loads the convention for a language and kind', () => {
    const text = loadConvention(byId('ts')!, 'unit')
    expect(text).toContain('Where it lives')
    expect(text).toContain('What to mock')
  })

  it('every language that canFix has a convention for all three kinds', () => {
    for (const lang of LANGUAGES.filter((l) => l.canFix)) {
      for (const kind of KINDS) {
        expect(() => loadConvention(lang, kind), `${lang.id}/${kind} is missing`).not.toThrow()
      }
    }
  })

  it('fails loudly when the convention does not exist', () => {
    const fake = { ...byId('ts')!, id: 'cobol' }
    expect(() => loadConvention(fake, 'unit')).toThrow(/no convention for cobol\/unit/)
  })
})
```

The second test is the important one: **it is what stops a language from claiming `canFix: true` without a written standard.** Flipping `canFix` on in the registry without writing the markdown breaks the build. That is the enforcement the design asks for — the tool executes the standard, so no standard means no fixing.

- [ ] **Step 5: Run it and watch it fail**

Run: `npx vitest run test/conventions.test.ts`
Expected: FAIL — module not found.

- [ ] **Step 6: Implement the loader**

`src/conventions.ts`:
```ts
import { existsSync, readFileSync } from 'node:fs'
import { dirname, join } from 'node:path'
import { fileURLToPath } from 'node:url'
import type { Language } from './languages.js'
import type { TestKind } from './types.js'

const PACKAGE_ROOT = join(dirname(fileURLToPath(import.meta.url)), '..')

/** The convention IS the specialist agent. No convention, no fixing — by design. */
export function loadConvention(lang: Language, kind: TestKind, root = PACKAGE_ROOT): string {
  const path = join(root, 'conventions', lang.id, `${kind}.md`)
  if (!existsSync(path)) {
    throw new Error(
      `redbar: no convention for ${lang.id}/${kind}. Write conventions/${lang.id}/${kind}.md — ` +
        `the standard is the deliverable, not the tool.`,
    )
  }
  return readFileSync(path, 'utf8')
}
```

- [ ] **Step 7: Run it and watch it pass**

Run: `npx vitest run test/conventions.test.ts && npx tsc --noEmit`
Expected: 3 passed. If the second test fails, a convention file is missing — write it, do not weaken the test.

git commit -m "feat: house testing conventions and loader"

### Task 2: `agents.ts` — the multi-LLM adapter

**Files:**
- Create: `src/agents.ts`
- Test: `test/agents.test.ts`

**Interfaces:**
- Produces:
```ts
export type Agent = { name: string; cmd: string; args: (prompt: string) => string[] }
export const AGENTS: Agent[]
export function resolveAgent(root: string, path?: string): Agent  // config → auto-detect → throw
export function runAgent(agent: Agent, prompt: string, cwd: string): { ok: boolean; output: string }
```

**The worker never talks to an LLM API.** It spawns the agent CLI the developer already has. Zero SDK, zero API key management, zero new cost — it rides the Copilot licence the company already pays for.

**Headless flags, checked at the source (`--help` on each CLI), not guessed:**

```ts
export const AGENTS: Agent[] = [
  { name: 'claude', cmd: 'claude', args: (p) => ['-p', p, '--permission-mode', 'acceptEdits'] },
  { name: 'codex', cmd: 'codex', args: (p) => ['exec', p] },
  { name: 'copilot', cmd: 'copilot', args: (p) => ['-p', p, '--allow-all-tools'] },
]
```

**Resolution order:**
1. `redbar.config.json` → `{"agent": "copilot"}` → that one, by name.
2. `redbar.config.json` → `{"agent": {"cmd": "anything", "args": ["--prompt", "{{prompt}}"]}}` → a custom command; `{{prompt}}` is substituted in every arg. **This is the escape hatch, and it is what keeps the tool alive**: it covers the company's internal gateway, a CLI that does not exist yet, and whatever model ships next month. Adding an LLM is one line of config, not a release.
3. No config → **auto-detect**: the first CLI found on `PATH` wins, in `AGENTS` order.
4. None found → throw, listing the supported ones and how to configure a custom one.

**Test with an injected `which`** (a `(cmd: string) => boolean` predicate) so the suite does not depend on which CLIs happen to be installed on the machine running it. `runAgent` gets its own test against a trivial fake command (`{"cmd": "echo", "args": ["{{prompt}}"]}`) — that proves substitution and the spawn path without needing a real agent.

- [ ] **Step 1: Write the failing test** — cases: config by name; config with a custom `cmd`/`args` and `{{prompt}}` substitution; auto-detect picks the first on `PATH`; auto-detect respects `AGENTS` order when several exist; throws with a helpful message when nothing is found; `runAgent` substitutes and returns the output.
- [ ] **Step 2: Run it and watch it fail.**
- [ ] **Step 3: Implement** using `node:child_process` `spawnSync` (`encoding: 'utf8'`, a generous `timeout`, `cwd` set to the target repo).
- [ ] **Step 4: Run it and watch it pass.**
- [ ] **Step 5: Commit** — `feat: multi-llm agent adapter`

### Task 3: `prompt.ts` + `testpath.ts` — composing the specialist

**Files:**
- Create: `src/prompt.ts`, `src/testpath.ts`
- Test: `test/prompt.test.ts`, `test/testpath.test.ts`

**Interfaces:**
- Produces:
```ts
export function testPathFor(gap: Gap, lang: Language): string
export function buildPrompt(gap: Gap, lang: Language, source: string, testPath: string): string
```

**`buildPrompt` is a pure string function, and the specialist agent lives entirely inside it:**

```
<convention>          ← conventions/<lang>/<gap.kind>.md, verbatim
<the gap>             ← file, symbol, uncovered lines, branch count
<the source>          ← the symbol's code, with line numbers
<the instruction>     ← write exactly one test file at <testPath>; make it pass; touch nothing else
```

The instruction block is explicit about the boundaries, because an agent given a free hand will refactor the source it was asked to test:

- Write **one** file, at exactly `<testPath>`. Create no other file.
- **Do not modify the source file.** If the code looks wrong, say so in a comment in the test — do not fix it.
- Cover the uncovered lines listed, especially every branch.
- The test must pass against the code **as it is today**.

**`testPathFor`** derives the test location from the convention's path rule — one line of data per language, in the registry (`testPath: (src, kind) => string`). Extend `Language` with it. Still no `switch` anywhere.

- [ ] **Step 1–5:** failing tests, run, implement, run, commit — `feat: prompt builder and test path resolution`

### Task 4: `fix.ts` — the worker pool and the quality gate

**Files:**
- Create: `src/fix.ts`
- Test: `test/fix.test.ts`

**Interfaces:**
- Produces:
```ts
export type FixResult = {
  gap: Gap
  testPath: string
  status: 'passed' | 'needs-human'
  attempts: number
  error?: string
}
export function fix(root: string, gaps: Gap[], opts?: { agent?: Agent; concurrency?: number }): Promise<FixResult[]>
```

**Partition by target file — this is the whole concurrency design.** Gaps are grouped by the test file they would write to, and each group goes to exactly one worker. Two workers can never write the same file, so a collision is impossible *by construction*. No worktrees, no locks, no merge step. The problem was eliminated in the design, not solved with a mutex — which is why the design explicitly rules git worktrees out of scope.

**The quality gate, per worker:**
1. Write the test (spawn the agent).
2. Run **only the test it just wrote** — not the suite. (`vitest run <path>`, `mvn test -Dtest=<Class>`, `pytest <path>`, `cargo test <name>`, `phpunit <path>` — one more line of data in the registry: `runOneTest: (path) => string`.)
3. Green → it goes in the report.
4. Red → **one** retry, with the failure output pasted into the prompt.
5. Red again → `needs-human`, and **the test file is deleted.** Never commit red. A red test leaking into the demo erases everything else.

Concurrency defaults to the core count (`os.cpus().length`).

**Testing this without burning tokens:** inject a fake agent (`{cmd: 'cp', args: [...]}` that copies a prepared fixture into place) and a fake test runner. The cases that matter: all green; one red that goes green on retry; one red twice → `needs-human` **and the file is gone from disk**; two gaps in the same file → one worker, not two.

- [ ] **Step 1–5:** failing tests, run, implement, run, commit — `feat: worker pool with quality gate`

### Task 5: `init.ts` — propose the libraries, let the human approve

**Files:**
- Create: `src/init.ts`
- Test: `test/init.test.ts`

**Interfaces:**
- Produces:
```ts
export type InitPlan = {
  language: Language
  missing: Record<TestKind, string[]>  // libs not yet in the manifest
  installCommand: string | null        // null when nothing is missing
  coverageCommand: string
}
export function planInit(root: string): InitPlan
export function renderInit(plan: InitPlan): string  // what gets printed
```

**It detects the language, reads the manifest, works out which of the registry's `testLibs` are already there, and prints only what is missing** — plus the exact command to install it and the coverage command to run afterwards.

**It never runs the install.** `planInit` is pure analysis; `renderInit` returns a string. The CLI prints it and stops. The human copies the command, or does not. This is the design's hard line and it is not negotiable: a tool that edits someone's `pom.xml` unasked is a supply-chain incident.

The output is a diff-shaped block, so it reads like a proposal rather than an announcement:

```
redbar init — TypeScript detected (vitest)

  test libraries missing:
  + @vitest/coverage-v8      (unit)
  + @playwright/test         (e2e)

  run this to add them:
    npm install -D @vitest/coverage-v8 @playwright/test

  then generate coverage with:
    npx vitest run --coverage
```

- [ ] **Step 1–5:** failing tests, run, implement, run, commit — `feat: init proposes test libraries for approval`

### Task 6: `report.ts` — the two audiences

**Files:**
- Create: `src/report.ts`
- Test: `test/report.test.ts`

**Interfaces:**
- Produces:
```ts
export function writeGapsJson(root: string, inspection: Inspection): string  // .redbar/gaps.json
export function renderMarkdown(inspection: Inspection, results?: FixResult[]): string  // REDBAR.md
```

Two audiences, two formats, one source of truth:

- **`.redbar/gaps.json`** — for the agent. The raw `Gap[]`, plus the language and base. Stable and machine-shaped.
- **`REDBAR.md`** — for the human and the PR. The ranked table (score, kind, file, symbol, lines, branches), and after a `fix` run: what was created, which convention it followed, and why that convention is right for that case. **This is the piece that closes the demo's narrative: hole → filled → justified.**

Both are deterministic: same input, byte-identical output. A report that reshuffles between runs cannot be diffed in a PR.

- [ ] **Step 1–5:** failing tests, run, implement, run, commit — `feat: gaps.json and REDBAR.md reports`

## What this plan does not cover

- `cli.ts` (`inspect | fix | init | ci`), `mcp.ts`, and the CI gate. They are thin shells over `inspect()` and `fix()` — days 14–15, and fast precisely because the engine and the agents already exist.
- `e2e.ts`: routes ∩ existing Playwright specs. `classify` already tags a gap as `e2e`; what is missing is crossing it against the specs that already exist, so redbar does not ask for a test that is already written.

## The risk to keep in view

The conventions documents are what the tool executes, so a bad convention is a bad tool. The failure mode is no longer "the team never agreed" — the standards ship with redbar and no meeting is required. The failure mode is now **inventing guidance and calling it a standard.**

If a rule in `conventions/` cannot be traced to the library's own documentation, it is somebody's opinion wearing a citation's clothes, and it forfeits the one thing that makes this work: **nobody argues with the Playwright docs.**

So the review gate on Task 1 is narrow and strict: **open the linked source, and check the rule is actually there.**
