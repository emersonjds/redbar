# redbar — Design

**Date:** 2026-07-12
**Context:** internal hackathon. 1 month deadline, targeting completion in 15–20 days to leave time for testing with the team, documentation and slides.

## Problem

There is no testing standard in the company. Everyone writes tests their own way, or doesn't write them at all. Whoever wants to test doesn't know **what** to test, and whoever uses an AI agent to write tests gets a test that follows the model's standard, not the house standard.

Two distinct gaps, and confusing them is the common mistake:

1. **Nobody knows where the holes are.** That is a *data* problem, not an AI problem.
2. **When the test does get written, it comes out off-standard.** That is a *convention* problem, not a tooling problem.

redbar attacks both, but with different mechanisms: the first with the compiler and git, the second with a conventions document the agent is forced to read.

## Core principle

**One engine, three consumers.**

The engine is a set of pure functions: detect stack → read coverage → cross with the diff → produce ranked gaps. CLI, MCP and CI gate are thin shells around it.

That is what keeps the project from turning into three projects, and it is what allows publishing it as a public skill later without rewriting anything — the packaging changes, not the core.

**`inspect` is zero-LLM.** No model is called during the analysis phase. The gap report is not an AI's opinion: it is the coverage report and `git diff` talking. That is what gives the number its authority and it is the central argument of the pitch.

Only `fix` touches an agent, and it touches it through a small adapter.

## Architecture

```
redbar/
  src/
    detect.ts          # package.json | pom.xml | pyproject.toml → stack + existing runner
    coverage/
      lcov.ts          # JS/TS  (Vitest, Jest)
      jacoco.ts        # Java   (JaCoCo XML)
      cobertura.ts     # Python (coverage.py XML)
    gap.ts             # git diff main...HEAD ∩ uncovered lines → ranked gaps
    e2e.ts             # routes ∩ Playwright specs → e2e gaps
    report.ts          # gaps.json (agent) + REDBAR.md (human)
    agents.ts          # multi-LLM adapter
    fix.ts             # worker pool
    conventions/       # the company's testing standard, one .md per stack
    cli.ts             # inspect | fix | init | ci
    mcp.ts             # the same engine exposed as MCP
  fixtures/
    ts/  spring/  py/  # minimal repos with a planted hole, to test the engine
```

`gap.ts` and `coverage/*` know nothing about the CLI, MCP or child processes. They are input → output.

## Flow

### 1. `redbar init`

Detects the stack and the runner the project already has. Prints the **diff** of the recommended setup (dependencies + coverage config) and asks for confirmation.

**It never installs a dependency on its own.** A tool that touches someone else's `pom.xml` or `package.json` without asking is a rejected PR in the best case and a supply chain incident in the worst. The human hits enter.

### 2. `redbar inspect` (`/inspect`)

1. Runs the coverage command the project already has.
2. Parses the report in the ecosystem's standard format (lcov / JaCoCo XML / Cobertura XML).
3. Crosses it with `git diff main...HEAD`: what **changed** and is **uncovered**.
4. Ranks by exported/public symbol with zero covered lines.
5. Enumerates e2e gaps: routes (`app/`, `pages/`, `@RestController`, `@RequestMapping`) with no Playwright spec touching them. Unit coverage cannot see this hole — it has to be its own step.
6. Writes `.redbar/gaps.json` (for the agent) and `REDBAR.md` (for the human and for the PR).

Runs in seconds. No AI.

### 3. `redbar fix` (`/fixit`)

Reads `gaps.json`, spins up a worker pool (N = number of cores). **Each worker takes 1 gap and writes 1 test file.**

**Partitioning by target file:** two workers never write to the same file. Collision is impossible by construction — no worktree, no lock, no merge. That is the reason there is no git isolation here: the problem was eliminated in the design, not solved with a lock.

**Quality gate, per worker:**
1. Writes the test.
2. Runs **only the test it wrote**.
3. Passed → goes into the final report.
4. Failed → one retry, with the error in the prompt.
5. Failed again → marked `needs-human` in the report.

**It never leaves a broken test in the repository.** One red test leaking into the demo erases everything else.

### 4. Final report

What was created, which standard was followed, and why that standard is the right one for that case. It is the piece that closes the demo's narrative: hole → plugged → justified.

## Multi-LLM

The worker **does not talk to any LLM API.** It `spawn()`s the agent CLI the person already has installed.

```ts
type Agent = { cmd: string; args: (prompt: string) => string[] }
```

Supported out of the box: **Claude Code**, **Codex**, **Copilot**. Configuration in `redbar.config.json`:

```json
{ "agent": "copilot" }
```

With no config: **auto-detect** — the first CLI found on `PATH` wins.

**Escape hatch**, and it is what keeps the tool from dying:

```json
{ "agent": { "cmd": "qualquer-coisa", "args": ["--prompt", "{{prompt}}"] } }
```

That covers the company's internal gateway, a CLI that doesn't exist yet, and the model that ships next month. Adding an LLM is one line of config, not a release.

Consequences that matter: **zero SDK, zero API key management, zero new cost** — it uses the Copilot license the company already pays for.

> **Open item:** each CLI's headless mode flags change every release. Check at the source (`--help` for each one) before writing the adapter. Don't guess.

On the MCP side, portability is free: Claude Code, Copilot agent mode and Codex all three speak MCP. Same engine, same tools, three clients, zero extra code.

## Adoption — the point most people get wrong

**You don't force an agent to use a tool. You force the PR.**

Three layers, in order of strength:

1. **CI gate** — the PR fails if new/changed code has a gap above the threshold. Runs the same binary as `inspect`. It is the only layer nobody works around.
2. **Config in the repository** — `AGENTS.md` and `.github/copilot-instructions.md` pointing at the same conventions document. The agent loads it on its own, without anyone installing anything.
3. **MCP** — convenience. The agent calls it and already knows what to do.

MCP is what makes it pleasant. CI is what makes it mandatory.

## Expected errors

| Situation | Behavior |
|---|---|
| Project with no coverage configured | `inspect` fails loudly, with the exact `init` command |
| Agent CLI not found | Clear message listing the supported ones and how to configure |
| Test doesn't pass even on the retry | `needs-human` in the report. Never committed |
| Repository with no `main` branch | Diff base is configurable; default detected from the remote |

## How redbar is tested

Three minimal fixture repositories (`fixtures/ts`, `fixtures/spring`, `fixtures/py`), each with a coverage hole planted on purpose. The test asserts that `gap.ts` finds exactly that hole.

One check that actually runs. Not a suite per function.

## Scope

| Stack | `inspect` (finds gaps) | `fix` (writes tests) |
|---|---|---|
| TS/JS (Vitest/Jest + Playwright) | Yes | Yes |
| Java / Spring Boot (JaCoCo) | Yes | Yes |
| Python (coverage.py) | Yes | **No** |

**Python does not write tests.** It's "a few scripts", it's not where the pain is, and it's the first candidate for the chop if the schedule tightens.

## Schedule

| Days | Deliverable |
|---|---|
| 1–4 | **Engine.** `detect` + 3 parsers + `gap.ts`. No AI, no pretty CLI. |
| 5–8 | **`/inspect`.** CLI, markdown report, `redbar init`, e2e gaps. |
| 9–13 | **`/fixit`.** Worker pool, multi-agent adapter, verification loop, final report. |
| 14–15 | **MCP + CI gate.** Thin shells — fast, because the engine already exists. |
| 16–20 | **Adoption.** Conventions doc, run it on 3–4 real repos, 2–3 people testing, slides. |

**Day 4 checkpoint:** run it on a real company repository and see the correct gaps come out. If that doesn't happen on day 4, everything else is fantasy — and there are still 16 days to correct course.

## Risks

**The conventions document is the only part that cannot be generated on its own.** It is *this* company's testing standard, and it needs people from the team weighing in. It is also the real deliverable — the tool merely executes it. **Start that conversation on day 1, not on day 16.**

**Real legacy repositories will break the engine.** Monorepo, badly configured coverage, generated code, `target/` full of junk. That's where 4 days get lost, and that's where the demo proves itself. The extra time in the schedule exists for that — not for more features.

## Out of scope

- Automatic dependency installation (`init` proposes, the human approves).
- Isolation via git worktree (eliminated by partitioning per file).
- Our own LLM SDK, API key management, new inference cost.
- Writing tests in Python.
